# Content Architecture

공개 표면의 역할, 연결 관계, 콘텐츠 배치 원칙을 정의하는 청사진.

각 표면의 실제 업데이트는 이 문서를 기준으로 이슈 사이클에서 진행한다.

## 정체성 (Identity)

- **한 줄 소개**: Spring Boot · Kubernetes 기반 멀티테넌시 서비스를 개발·운영하는 7년차 백엔드 개발자
- **핵심 키워드**: 백엔드, Kubernetes, 멀티테넌시, 과금 미터링 데이터 파이프라인, 서비스 개발·운영, 인프라 협업, 인증서 자동화
- **내러티브**: 과금 미터링 데이터 파이프라인을 설계해 일 수십만 건을 수집·집계하고, 서비스 개발과 운영을 함께 맡아 리소스·노드 안정화 같은 운영 이슈를 인프라와 함께 풀어왔습니다. 마주친 결정과 공부한 것을 블로그에 기록합니다.

### 본질·부가 위계

- **본질**: 백엔드 개발·운영 (Spring Boot · Kubernetes · 멀티테넌시 서비스). 미터링 파이프라인 설계, 서비스 개발과 운영을 함께 맡은 인프라 협업이 어필 축.
- **부가**: 개인 개발 운영 하네스(claude-ops-agent)는 개인 프로젝트 위치에서만 노출. SDD·ADR·추상 설계(DDD·헥사고날)는 자기소개 한 줄·허브에서 비노출 (Tech Stack 뱃지는 이력서 정합용 검증 가능한 사실이라 예외).

자기소개·정체성 표면에서 메타-역량(트레이드오프·SDD·ADR) 자기 평가형 표현은 회피한다. 인터미디에이트 7년차 레벨에 부담이며, 면접관(특히 대기업·대용량 트래픽 기업) 시각에서 본질 직무 정체성을 가린다.

추상 설계(DDD·헥사고날)는 면접관 취향에 따라 정답이 갈려 꼬리질문 공격면을 넓힌다. 공개 표면에서 키워드로 내세우지 않고, 필요 시 방어 가능한 깊이까지만 (career-docs 선임 조언 라운드 항목 3, ADR-036). 기술 선택 서사는 보편·표준 기준을 우선으로 서술한다 (동 라운드 항목 5).

### Tech Stack 뱃지 예외 원칙 (본질·부가 carve-out)

"AI 도구는 부가, 자기소개 한 줄에서 제외"가 본질·부가 위계의 기본이다. Tech Stack 뱃지는 이 위계의 명시적 예외다. 예외가 개별 결정으로 축적되며 경계가 흐려지지 않도록 근거·경계를 고정한다.

- **예외 근거**: 뱃지는 자기 평가가 아니라 검증 가능한 사실 나열이다. 이력서 skills 섹션과 1:1로 정합하면 "부가 어필"이 아니라 "사용 기술 명시"로 읽힌다. 그래서 부가(AI 도구)라도 뱃지에는 오를 수 있다.
- **적용 경계 (무엇이 뱃지에 오를 수 있는가)**:
  1. 이력서 skills 섹션에 있는 기술만. 이력서에 없는 기술은 뱃지 금지.
  2. 실제 사용한 검증 가능한 기술만 (career-docs ADR-032 사실 보호 · spec-11 미경험 기술 미기재).
  3. 뱃지는 나열까지만. 숙련도·자기 평가 문구를 덧붙이지 않는다.
- **AI 섹션 확장 기준**: AI 뱃지(Claude Code·MCP 등)는 위 경계를 따르되, 개수·비중이 Backend·Infra 카테고리를 넘지 않는다. AI를 별도 강조 카테고리로 키우지 않는다 (본질=백엔드 유지). 새 AI 뱃지 추가는 이력서 skills 반영이 선행 조건.

## 표면 정의 (Surface Map)

### 비공개 표면

채용팀·헤드헌터에만 노출. 이직 의도 키워드 명시 가능.

| 표면 | 위치 | 역할 |
|------|------|------|
| 이력서 / 포트폴리오 | career-docs (비공개 레포) | 풀 노출, 채용팀 직접 대상 |
| 신청서 / 커피챗 | 각 공고 양식 | 풀 노출 + 지향점 명시 |

**비공개 이직 방향** (career-docs `profile/career-context.md` SSOT): 지금(7년차)은 대규모 트래픽 경험 확보가 최우선. 트래픽 규모가 크고 기반이 탄탄한 회사, 시니어가 있는 팀에서 인터미디에이트로 트래픽을 겪을 수 있는 곳. 회사 유형은 서비스 1순위·금융 2순위 (선임 조언 라운드 항목 10). 공개 표면에는 이 방향을 노출하지 않는다.

### 공개 표면

누구나(현 회사 동료·상사 포함) 접근 가능. **이직 의도 키워드 금지**.

| 표면 | 위치 | 역할 | 대상 |
|------|------|------|------|
| **GitHub Profile README** | [idean3885/idean3885](https://github.com/idean3885/idean3885) | 허브 - 직무 정체성 + 운영 사례·개인 프로젝트 진입점 | GitHub 방문자 |
| **Blog Repo About** | blog.idean.me repo settings | 메타데이터 - 검색·탐색 노출 | GitHub 검색 |
| **Blog About (홈 소개 섹션)** | [홈 페이지 소개 영역](https://blog.idean.me) | 깊이 - 자기소개, 포커스 영역 안내 | 블로그 방문자 |

표면 분리 정책 상세는 career-docs ADR-016 참조.

## 정보 흐름 (Information Flow)

```
GitHub Profile README (허브)
  ├──→ Blog (블로그 링크)
  └──→ Personal Projects (개인 프로젝트 링크)

Blog 홈 소개 섹션 (깊이)
  ├──→ 포커스 영역별 포스트 링크
  └──→ 주요 포스팅 링크

Blog Repo About (메타데이터)
  └──→ Blog homepage URL
```

**원칙**: 허브(Profile README)에서 스포크(블로그, 개인 프로젝트)로 연결한다.

깊이 있는 내용은 각 스포크에 두고,
허브는 요약과 링크만 유지한다.

### 배치 기준

1. **본질 우선**: 한 줄 소개·주요 글은 백엔드 개발·운영 사례 중심.
   미터링 파이프라인 설계, 서비스 개발·운영을 함께 맡은 인프라 협업이 어필 축.
   AI 도구·추상 설계는 부가로 분리.
2. **검증 가능한 약속만**: 자기 평가형 메타 표현(트레이드오프·정량 의사결정) 회피.
   구체적 사실(서비스명·기술 스택·운영 영역)로 진술.
3. **추상 설계·비표준 기술 노출 절제**: DDD·헥사고날은 키워드로 내세우지 않는다.
   기술 선택은 보편·표준 기준을 우선으로 서술하고,
   비표준 선택은 근거(왜 그 선택이었나)를 방어 가능한 깊이까지만.
4. **중복 최소화**: 같은 정보를 여러 곳에 두지 않는다.
   한 곳에 원본을 두고 나머지는 링크로 연결.
5. **깊이 분리**: 허브는 요약, 스포크는 상세.
   Profile README에 포스트 전체 목록을 넣지 않는다.
6. **큐레이션 우선**: 주요 글은 대표성 있는 운영 사례만 수동 관리.
   메타-역량 자기 평가형 글(방법론·회고·도구 소개)은 회피.
   블로그 자체 글 정리(예정) 전까지 보수적 유지.
7. **이직 의도 누출 금지**: 공개 표면 어디에도 "이직 준비", "Open to work", "다음 단계로 ~ 기여" 같은 키워드 X.

### 개인 프로젝트 노출 정책

- 섹션명은 이력서와 동일하게 **개인 프로젝트**로 통일 (career-docs resume 정합)
- 본업(백엔드) 외 부수 위치로 노출하고 섹션 제목·부제("백엔드 본업 외…")로 가중치를 명시 (면접관이 본업과 혼동하지 않도록)
- **메인 = claude-ops-agent**: 이슈 플로우·콘텐츠 작성·멀티레포 조정·검증을 하나로 묶은 개인 개발 운영 하네스. 초기 cross-verify 단독 노출은 ops-agent(구 devex) 내장 기능으로 흡수됨 (career-docs ADR-0047·0050). AI 사용이 기본이 된 환경에서 차별 축은 "개인 워크플로우 통합·멀티레포 오케스트레이션"
- AI 엔지니어링(모델 학습·RAG·Agent 설계) 포장 금지. 도구 아키텍처 판단(분리 vs 통합)·워크플로우 자동화로 서술 (career-docs career-context 원칙)
- **trip-planner 표면 분리**: 이력서/포트폴리오(채용 패키지)에서는 제외 (career-docs ADR-0024: 결과 미완성·회사 구조 노출·기획자 대체 신호). 공개 허브(README)에는 외부 자산·풀스택 사이드로 유지 (ADR-0024가 외부 표면 유지 명시)
- 운영 중이거나 지속 관리하는 자산만 노출한다

### 2026-07-18 새 이력서·포폴 사상 정합 (청사진 재작성)

career-docs 이력서·포트폴리오 전면 재작성 과정에서 사상이 바뀌어, 뒤처진 이 청사진을 하위 표면·새 사상에 맞춰 재정합. 원천은 아래 career-docs 결정.

- **선임 조언 방향 전환 (2026-06-29, docs/revisions/2026-06-29-senior-advice-direction.md)**: 서류 통과 우선(스터디 후순위), 추상 설계(DDD·헥사고날) 노출 수위 하향, 보편·표준 기술 중시, 사고방식(Why·판단 과정) 어필, 도메인 깊이로 핏 보완, 서비스 1순위·금융 2순위. 이력서 백지 재작성 결정 — 서비스/데이터 BE 재조준 + 은행권 약점 교정 (전부 갈아엎기 아님).
- **한 줄·키워드 정합**: "클라우드 서비스 설계·운영" → "멀티테넌시 서비스 개발·운영". "시계열 파이프라인" → "과금 미터링 데이터 파이프라인". 서비스 개발과 운영을 함께 맡은 인프라 협업을 어필 축으로 전면화.
- **ops-agent 개인 프로젝트 메인 승격 (career-docs ADR-0047, 0050 rename)**: cross-verify 단독 → ops-agent(구 devex) 내장 흡수. 차별 축을 "개인 워크플로우 통합·멀티레포 오케스트레이션"으로. AI 엔지니어링 포장 금지 원칙 명시.
